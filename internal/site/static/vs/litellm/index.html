When to choose Stockyard

You want a single binary that handles everything — proxy, observability, audit trails, prompt management, and workflows — without standing up Redis, Postgres, or separate services. You value fast cold starts and low memory for edge or containerized deployments.

When to choose LiteLLM

You’re already invested in the Python ecosystem and need deep integration with Python-specific tools. You want a mature, widely-adopted proxy with a large community and are comfortable managing Redis and Postgres infrastructure.

The infrastructure gap

LiteLLM requires you to provision and manage Redis (for caching and rate limiting) and Postgres (for logging and config). Stockyard embeds SQLite, so there’s nothing to provision. Run stockyard and everything works. This matters most in environments where you can’t or don’t want to manage additional databases — local development, CI pipelines, edge deployments, or single-server setups.

Beyond the proxy

LiteLLM is a proxy — it routes requests and logs them. Stockyard is a platform. The same binary that proxies your requests also gives you a hash-chained audit ledger (Trust), versioned prompt templates with A/B experiments (Studio), a DAG workflow engine (Forge), and an installable config marketplace (Exchange). These aren’t separate services to deploy. They’re part of the binary.
